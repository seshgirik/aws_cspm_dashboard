üî¥ Public EKS Cluster Endpoint Without CIDR Restrictions
Impact: Kubernetes API server accessible from internet. Cluster takeover, pod compromise, data access.

üî¥ S3 Bucket Exposed With Sensitive API Credentials
Impact: AWS access keys, API tokens publicly readable. Account takeover, resource manipulation, data breach.
Fix: Remove public access, rotate all exposed credentials immediately, use IAM roles instead.
üî¥ Container Using hostPID, hostIPC, hostNetwork
Impact: Container can see all host processes, access host network. Container escape, host compromise.

üî¥ Network Path Allows Exposure to Internet
Impact: Internal resources reachable from internet via misconfigured routing or NAT.
Fix: Review VPC route tables, implement proper subnet isolation, use Network Firewall.
üî¥ Flat Network Architecture - No Microsegmentation
Impact: All services can communicate freely. Lateral movement, blast radius amplification.
Fix: Implement security groups per tier, use Network ACLs, deploy service mesh.
üî¥ No Kubernetes NetworkPolicies Defined
Impact: Any pod can talk to any pod. Compromised pod can pivot to entire cluster.
Fix: Implement NetworkPolicy objects, default deny all, explicit allow rules.

üë§ IAM Overprivileges & Admin Access

23 CRITICAL
üî¥ IAM Role with Wildcard Permissions (Action: *, Resource: *)
Impact: Full AWS account control. Can delete all resources, create backdoors, disable logging.
Fix: Apply least privilege. Use IAM Access Analyzer to identify actual permissions needed.
üî¥ IAM Root Account Access Keys Exist
Impact: Root keys have unrestricted access, cannot be limited by policies. If leaked = total compromise.
Fix: Delete root access keys immediately. Use IAM users/roles with MFA for all access.
üî¥ EC2 Instance Profile with AdministratorAccess Policy
Impact: If EC2 compromised, attacker has full AWS access. SSRF attacks become catastrophic.
Fix: Create minimal custom policy. Use aws:SourceIp conditions. Enable IMDSv2.
üî¥ IAM User with Privilege Escalation Permissions
Impact: Can create new admin users, attach policies to self. Bypasses all access controls.
Fix: Remove CreateUser, AttachUserPolicy, PutUserPolicy permissions. Audit regularly.
üî¥ IAM Role Trust Policy Allowing Any AWS Account (Principal: *)
Impact: Any AWS account can assume this role. Unauthorized cross-account access.
Fix: Restrict Principal to specific AWS account IDs. Add external ID for third parties.
üî¥ IAM Policy with CloudTrail/CloudWatch Logs Deletion
Impact: Attacker can delete audit logs, hide their tracks. Forensics impossible.
Fix: Remove delete permissions. Use S3 Object Lock for CloudTrail logs.
üî¥ IAM Group with Destructive Permissions
Impact: Can DeleteBucket, DeleteDBInstance, TerminateInstances. Single mistake = disaster.
Fix: Require MFA for destructive actions. Implement approval workflows.
üî¥ Service Accounts Have Excessive Permissions
Impact: Automated systems with admin access. Compromised CI/CD = total breach.
Fix: Scope permissions to specific resources. Use OIDC federation for GitHub Actions.
üî¥ IAM User with Unrestricted KMS Decrypt & Secrets Manager
Impact: Can decrypt all encrypted data, read all secrets. Mass credential exposure.
Fix: Restrict to specific KMS keys/secrets. Use resource-based policies.
üî¥ Privileged Access Management Not Implemented - No JIT
Impact: Permanent admin access increases attack surface. Stolen credentials valid indefinitely.
Fix: Implement just-in-time (JIT) access with AWS SSO/IAM Identity Center.
